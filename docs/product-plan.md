# 제품 기획안

> 이 문서는 초안이다. 세부 수치나 지원 직무 문구처럼 본인만 알 수 있는 부분은 표시해뒀으니 확인
> 후 수정할 것.
>
> **2026-07-09 업데이트**: 화면 구성을 "캔버스 에디터 / 노드 그래프 / 3D 프리뷰" 3패널 탭
> 구조에서 **ComfyUI류의 단일 노드 파이프라인 캔버스**로 전면 수정했다. 이 문서(1~6번 섹션)는
> 새 구조를 반영해 다시 썼지만, 다음 문서들은 아직 옛 3패널 구조를 전제로 쓰여 있어 뒤따라
> 수정이 필요하다: `docs/architecture.md`("클라이언트" 구성 요소 설명), `docs/data-model.md`
> (노드가 하나의 스키마였는데 이제 타입별로 필드가 다름), `docs/acceptance-criteria.md`(스텁,
> 시나리오 번호가 옛 구조 기준), `feature_list.json`(`canvas-crud`/`node-graph`/`preview-3d`가
> 사실상 하나의 노드 캔버스 feature로 합쳐질 가능성이 큼).

## 1. 목적

개인 포트폴리오 프로젝트다. "실시간 협업이 되는 AI 노드 파이프라인 빌더"(ComfyUI류 워크플로우
캔버스)를 처음부터 끝까지 직접 설계·구현해서, 다음 세 가지를 증명하는 것이 목적이다.

1. CRDT(Yjs) 기반 실시간 협업 시스템을 프레임워크(Hocuspocus 등) 뒤에 숨지 않고 프로토콜
   레벨(`y-protocols/sync`, `y-protocols/awareness`)에서 직접 이해하고 구현할 수 있다.
2. 상태가 있는 WebSocket 서버(room 단위 in-memory 관리)와 상태 없는 REST API(외부 AI 이미지
   생성 API 프록시)를 한 시스템 안에서 역할을 분리해 설계할 수 있다.
3. React Three Fiber 없이 Three.js를 직접 다뤄서 노드 파이프라인의 3D 생성 결과를 렌더링하고,
   기본적인 렌더링 최적화(드로우콜/프레임 타임 개선)를 수치로 보여줄 수 있다.

해결하려는 문제(제품 관점의 사용자 문제)는 부차적이다 — "여러 명이 동시에 하나의 AI 생성
파이프라인(텍스트 → 이미지 → 3D)을 조립하고, 분기해서 여러 결과를 한 화면에서 비교하는" 흐름
자체가 지금은 여러 도구(프롬프트 관리 + 이미지 생성 툴 + 3D 뷰어)를 오가야 하는데, 이걸 하나의
웹 툴로 붙여본다는 데모다.

## 2. 사용자 시나리오

**대상 사용자**: 실제 서비스 사용자가 아니라, 이 프로젝트를 평가하는 사람(채용 담당자/면접관) 또는
직접 써보는 소규모 팀(2~3명이 같은 링크로 들어와 함께 파이프라인을 만드는 상황)을 가정한다.

**대표 흐름**:

1. 사용자 A가 room 링크(`?room=<id>`)로 접속한다. 빈 노드 캔버스가 보인다.
2. **Text Prompt** 노드를 캔버스에 추가하고 프롬프트를 입력한다.
3. **Generate Image** 노드를 여러 개 추가하고, Text Prompt 노드의 출력 포트를 각 Generate
   Image 노드의 입력 포트에 연결한다(하나의 출력에서 여러 엣지로 분기 — fan-out). 각 Generate
   Image 노드를 실행하면 서버가 외부 AI 이미지 생성 API를 대신 호출하고, 결과 이미지가 해당
   노드에 표시된다.
4. 마음에 든 이미지 노드 중 하나(또는 여러 개)의 출력을 **Generate 3D** 노드의 입력에 연결한다.
   Generate 3D 노드를 실행하면 이미지가 3D 결과로 변환되고, 노드 안(또는 확대 뷰)에서
   OrbitControls로 회전하며 확인할 수 있다.
5. 사용자 B가 같은 room 링크로 접속한다 → A가 만든 노드/엣지 그래프를 실시간으로 그대로 본다 →
   B가 새 노드를 추가하거나 다른 이미지 노드에서 새로운 Generate 3D 노드로 엣지를 연결하면 A의
   화면에도 즉시 반영된다.
6. 같은 이미지 노드 출력에서 다시 다른 파라미터의 Generate 3D 노드로 분기해, 여러 3D 결과를 한
   화면에서 나란히 비교한다.

이 흐름은 `docs/acceptance-criteria.md`(스텁, 아직 옛 구조 기준 시나리오만 있어 다시 써야 함)와
매핑되어야 한다.

## 3. 화면 구성

**단일 화면, 단일 캔버스.** 옛 기획의 "캔버스 에디터 / 노드 그래프 / 3D 프리뷰" 3패널 탭 구조는
버린다. 캔버스 자체가 곧 노드 파이프라인 에디터(React Flow 기반)다 — 별도의 자유 배치 이미지
캔버스나 별도의 3D 프리뷰 탭이 없다.

### 노드 타입 (3종)

| 노드 타입 | 입력 포트 | 출력 포트 | 동작 |
| --- | --- | --- | --- |
| **Text Prompt** | 없음 | text 1개 | 사용자가 직접 텍스트를 입력하는 시작 노드 |
| **Generate Image** | text (1개 이상) | image 1개 | 입력받은 텍스트로 REST API를 호출해 이미지 생성 |
| **Generate 3D** | image (1개 이상) | (없음, 터미널 노드) | 입력받은 이미지로 3D 결과 생성, 노드 내부에서 Three.js로 렌더링 + OrbitControls |

### 연결(엣지) 규칙

- 포트에는 타입이 있다: `text` 포트는 `text` 포트끼리만, `image` 포트는 `image` 포트끼리만
  연결된다(**text-to-text**, **image-to-image**). 타입이 다른 포트끼리는 연결할 수 없다.
- 한 출력 포트에서 여러 엣지로 분기(fan-out)할 수 있다 — 예: Text Prompt 하나의 출력을 여러
  Generate Image 노드에 동시에 연결.
- 한 입력 포트에 여러 엣지가 들어오는 것(fan-in, 예: Generate 3D 노드가 이미지 여러 개를 입력
  받아 조합)도 허용할지는 아직 미정 — `docs/data-model.md`를 다시 쓸 때 확정한다.
- 예시 파이프라인: `Text Prompt → Generate Image * n → Generate 3D * n` (이미지 노드 각각에서
  3D 노드로 개별 연결).

### 협업 동기화

노드/엣지 그래프 전체가 하나의 room의 하나의 Y.Doc을 공유한다. 화면이 하나뿐이므로(옛 기획처럼
"화면 전환은 로컬 UI 상태"라는 구분 자체가 필요 없어짐) 캔버스에 보이는 모든 것이 곧 동기화
대상이다.

## 4. 지원 직무와의 연결

> 본인이 실제로 지원하는 직무명/회사에 맞춰 이 절을 구체화할 것. 아래는 프로젝트가 이미 보여주는
> 역량을 기술 스택 기준으로 정리한 것이다.

- **실시간 협업/분산 상태 시스템**: CRDT 프로토콜(sync/awareness) 직접 구현, room 기반 상태
  관리, WebSocket 중계 서버 설계.
- **풀스택 TypeScript**: 모노레포(Turborepo)에서 타입을 프론트/백엔드가 공유(`packages/shared-types`),
  빌드/린트/테스트 파이프라인을 직접 구성.
- **API 설계와 보안 기본기**: 외부 API 키를 서버에서만 다루는 프록시 패턴, REST API 통합 테스트.
- **그래픽스/성능**: Three.js 직접 사용, 노드 기반 파이프라인의 3D 생성 결과 렌더링과 최적화를
  수치로 측정하고 기록하는 습관.
- **그래프 자료구조/에디터 UX**: 타입이 있는 포트, fan-out(및 잠재적 fan-in) 연결 규칙을 가진
  노드 에디터를 React Flow 위에서 설계.

## 5. 범위 조정 옵션

시간이 부족할 경우 아래 우선순위로 잘라낸다 (위쪽일수록 나중에 자름 = 더 핵심). 노드 캔버스가
하나로 합쳐지면서 옛 "캔버스 에디터 vs 노드 그래프"의 구분이 없어졌으므로, 아래 항목은
`feature_list.json`을 다시 쓸 때 이 기준으로 feature를 재구성해야 한다.

1. **반드시 남김**: 노드 캔버스 협업(Text Prompt/Generate Image 노드 + 연결 + Yjs 동기화) — 이
   것만 있어도 "실시간 협업 + AI 생성 파이프라인"이라는 핵심 데모가 성립한다.
2. **가능하면 남김**: 분기(fan-out) — 하나의 노드에서 여러 노드로 연결해 결과를 비교하는 것.
   없어도 파이프라인은 동작하지만, 차별점(여러 결과를 나란히 비교)이 사라진다.
3. **잘라도 되는 후보 1순위**: Generate 3D 노드 — 독립적인 기능이라 다른 기능에 영향을 주지 않고
   제외 가능(Text Prompt → Generate Image까지만으로도 데모 성립).
4. **잘라도 되는 후보 2순위**: 최적화 패스(`optimization-pass`) — 데모 자체는 최적화 없이도
   동작한다. 시간이 없으면 "적용 안 함, 향후 과제"로 남긴다.
5. **협업 기능 자체를 줄이는 경우**: awareness(커서/선택 상태 공유)부터 빼고 sync(그래프
   동기화)는 남긴다 — 협업의 핵심은 상태 동기화이지 커서 표시가 아니다.

## 6. Three.js 최적화 체크리스트

`optimization-pass` feature가 적용 여부를 기록해야 하는 항목. 우선순위 순으로 나열했고,
실제 적용 시 `feature_list.json`의 `optimization-pass.evidence`에 전/후 수치를 남긴다.

1. **텍스처 재사용/해제**: 3D 결과가 갱신될 때마다 새 텍스처를 만들지 않고
   `texture.needsUpdate = true`로 갱신. 사용하지 않게 된 텍스처는 `dispose()`.
2. **지오메트리/머티리얼 공유**: 여러 Generate 3D 노드가 동시에 화면에 있으면 지오메트리·머티리얼
   인스턴스를 공유해 드로우콜을 줄인다.
3. **불필요한 리렌더 방지**: `useFrame`(또는 애니메이션 루프)에서 React state를 직접 건드리지
   않고 ref로 처리해 React 리렌더와 렌더 루프를 분리한다.
4. **픽셀 비율 상한**: `renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))`로 고DPI
   화면에서의 과도한 해상도를 제한한다.
5. **프레임 측정**: `renderer.info.render.calls`(드로우콜)와 프레임 간 시간을 측정하는 간단한
   HUD 또는 콘솔 로그를 최적화 전/후 비교용으로 남긴다.

우선순위가 가장 높은 1~2번만 적용해도 벤치마크 비교 기록은 가능하다.
