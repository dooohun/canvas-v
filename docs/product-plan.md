# 제품 기획안

> 이 문서는 초안이다. `docs/architecture.md`, `CLAUDE.md`, `docs/acceptance-criteria.md`(스텁)에
> 이미 확정된 내용(기술 스택, Hocuspocus 미사용 결정, 절대 규칙)과 일관되게 작성했다. 세부 수치나
> 지원 직무 문구처럼 본인만 알 수 있는 부분은 표시해뒀으니 확인 후 수정할 것.

## 1. 목적

개인 포트폴리오 프로젝트다. "실시간 협업이 되는 AI 이미지 생성 툴"을 처음부터 끝까지 직접
설계·구현해서, 다음 세 가지를 증명하는 것이 목적이다.

1. CRDT(Yjs) 기반 실시간 협업 시스템을 프레임워크(Hocuspocus 등) 뒤에 숨지 않고 프로토콜
   레벨(`y-protocols/sync`, `y-protocols/awareness`)에서 직접 이해하고 구현할 수 있다.
2. 상태가 있는 WebSocket 서버(room 단위 in-memory 관리)와 상태 없는 REST API(외부 AI 이미지
   생성 API 프록시)를 한 시스템 안에서 역할을 분리해 설계할 수 있다.
3. React Three Fiber 없이 Three.js를 직접 다뤄서 캔버스 합성 결과를 3D 텍스처로 매핑하고,
   기본적인 렌더링 최적화(드로우콜/프레임 타임 개선)를 수치로 보여줄 수 있다.

해결하려는 문제(제품 관점의 사용자 문제)는 부차적이다 — "여러 명이 동시에 AI 생성 이미지를
가져와 하나의 씬으로 합성하고, 그 결과를 바로 3D로 확인하는" 흐름 자체가 지금은 여러 도구
(이미지 생성 툴 + 디자인 툴 + 3D 뷰어)를 오가야 하는데, 이걸 하나의 웹 툴로 붙여본다는 데모다.

## 2. 사용자 시나리오

**대상 사용자**: 실제 서비스 사용자가 아니라, 이 프로젝트를 평가하는 사람(채용 담당자/면접관) 또는
직접 써보는 소규모 팀(2~3명이 같은 링크로 들어와 함께 에셋을 만드는 상황)을 가정한다.

**대표 흐름**:

1. 사용자 A가 room 링크(`?room=<id>`)로 접속한다.
2. 프롬프트를 입력해 이미지를 생성한다 → 서버가 외부 AI API를 대신 호출하고 이미지 URL을
   반환한다 → 노드 그래프에 새 노드로 기록된다.
3. 생성된 이미지를 캔버스로 드래그해 오브젝트로 추가하고, 위치/크기/회전을 조정한다.
4. 사용자 B가 같은 room 링크로 접속한다 → A가 만든 캔버스 오브젝트와 노드 그래프를 그대로
   본다 → B가 오브젝트를 옮기면 A의 화면에도 실시간 반영된다.
5. 마음에 든 생성 결과에서 다른 프롬프트로 분기해 새로운 이미지를 생성한다 → 노드 그래프에
   새 엣지가 생긴다.
6. 캔버스 합성이 끝나면 3D 프리뷰 탭으로 전환해, 합성 결과가 텍스처로 입혀진 3D 모델을
   OrbitControls로 돌려본다.

이 흐름은 `docs/acceptance-criteria.md`의 시나리오 1~11과 1:1로 대응한다 (협업 1~5, 생성+그래프
6~8·10~11, 3D 9).

## 3. 화면 구성

단일 페이지, 3개 패널을 탭 또는 리사이즈 가능한 분할로 전환하는 구성을 기본으로 한다.

- **캔버스 에디터** (기본 화면): 이미지 레이어를 배치·편집하는 2D 캔버스. 상단에 프롬프트
  입력창.
- **노드 그래프**: React Flow 기반, 생성 히스토리를 노드/엣지로 표시. 노드를 클릭하면 해당
  생성 결과를 캔버스에 다시 가져올 수 있다.
- **3D 프리뷰**: 캔버스 합성 결과를 텍스처로 매핑한 3D 모델 + OrbitControls.

세 화면은 같은 room의 같은 Y.Doc을 공유하므로, 화면 전환은 클라이언트 로컬 UI 상태일 뿐 별도
동기화가 필요 없다.

## 4. 지원 직무와의 연결

> 본인이 실제로 지원하는 직무명/회사에 맞춰 이 절을 구체화할 것. 아래는 프로젝트가 이미 보여주는
> 역량을 기술 스택 기준으로 정리한 것이다.

- **실시간 협업/분산 상태 시스템**: CRDT 프로토콜(sync/awareness) 직접 구현, room 기반 상태
  관리, WebSocket 중계 서버 설계.
- **풀스택 TypeScript**: 모노레포(Turborepo)에서 타입을 프론트/백엔드가 공유(`packages/shared-types`),
  빌드/린트/테스트 파이프라인을 직접 구성.
- **API 설계와 보안 기본기**: 외부 API 키를 서버에서만 다루는 프록시 패턴, REST API 통합 테스트.
- **그래픽스/성능**: Three.js 직접 사용, 렌더링 최적화를 수치로 측정하고 기록하는 습관.

## 5. 범위 조정 옵션

시간이 부족할 경우 아래 우선순위로 잘라낸다 (위쪽일수록 나중에 자름 = 더 핵심).

1. **반드시 남김**: 협업 캔버스(`collab-canvas`) + AI 이미지 생성(`ai-image-generation`) — 이
   두 개만 있어도 "실시간 협업 + AI 생성"이라는 핵심 데모가 성립한다.
2. **가능하면 남김**: 노드 그래프(`node-graph`) — 없어도 생성은 되지만, 분기 탐색이라는
   차별점이 사라진다. 최소 버전으로는 "생성 히스토리 리스트"로 축소 가능.
3. **잘라도 되는 후보 1순위**: 3D 프리뷰(`preview-3d`) — 독립적인 기능이라 다른 기능에 영향을
   주지 않고 제외 가능.
4. **잘라도 되는 후보 2순위**: 최적화 패스(`optimization-pass`) — 데모 자체는 최적화 없이도
   동작한다. 시간이 없으면 "적용 안 함, 향후 과제"로 남긴다.
5. **협업 기능 자체를 줄이는 경우**: awareness(커서/선택 상태 공유, 시나리오 5)부터 빼고
   sync(문서 동기화)는 남긴다 — 협업의 핵심은 상태 동기화이지 커서 표시가 아니다.

## 6. Three.js 최적화 체크리스트

`optimization-pass` feature가 적용 여부를 기록해야 하는 항목. 우선순위 순으로 나열했고,
실제 적용 시 `feature_list.json`의 `optimization-pass.evidence`에 전/후 수치를 남긴다.

1. **텍스처 재사용/해제**: 캔버스가 갱신될 때마다 새 `THREE.CanvasTexture`를 만들지 않고
   `texture.needsUpdate = true`로 갱신. 사용하지 않게 된 텍스처는 `dispose()`.
2. **지오메트리/머티리얼 공유**: 3D 프리뷰에 오브젝트가 여러 개면 지오메트리·머티리얼 인스턴스를
   공유해 드로우콜을 줄인다.
3. **불필요한 리렌더 방지**: `useFrame`(또는 애니메이션 루프)에서 React state를 직접 건드리지
   않고 ref로 처리해 React 리렌더와 렌더 루프를 분리한다.
4. **픽셀 비율 상한**: `renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))`로 고DPI
   화면에서의 과도한 해상도를 제한한다.
5. **프레임 측정**: `renderer.info.render.calls`(드로우콜)와 프레임 간 시간을 측정하는 간단한
   HUD 또는 콘솔 로그를 최적화 전/후 비교용으로 남긴다.

우선순위가 가장 높은 1~2번만 적용해도 벤치마크 비교 기록은 가능하다.
